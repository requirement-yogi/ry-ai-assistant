# RY AI Assistant — MCP Server

## What this project does

An MCP (Model Context Protocol) server in TypeScript that lets an LLM client (Claude Desktop, Claude Code, Cursor…) transform a user prompt into a structured requirements tree in JSON format, refine it through conversation, and send it to a backend that generates a specification page (Confluence or other).

**The client LLM does the decomposition work** — the MCP server provides the schema, validates the JSON (Zod), and sends to the API. No server-side LLM calls.

## Stack

- TypeScript + Node.js (ESM)
- `@modelcontextprotocol/sdk` for the MCP server
- `zod` v4 for schema validation
- Transport: stdio (standard MCP)

## Commands

```bash
npm run build   # compile TypeScript → dist/
npm run dev     # run in dev mode with tsx (no build needed)
npm start       # run the compiled build
npm run mock    # start a local mock API server on :3000
```

## Architecture

```
src/
├── index.ts                  # MCP server, registers the 4 tools
├── schemas/requirements.ts   # Zod schema + TypeScript types
└── tools/
    ├── analyze.ts            # analyze tool constants
    ├── refine.ts             # refine tool constants
    ├── render.ts             # JSON → Markdown conversion (renderTree)
    └── submit.ts             # POST Markdown to the backend API (mock for now)
```

## The 4 MCP Tools

| Tool | Input | Role |
|---|---|---|
| `analyze_prompt` | `prompt`, `project_name` | Instructs the LLM to produce the initial JSON tree |
| `refine_requirements` | `current_tree`, `feedback` | Validates the incoming tree then instructs the LLM to apply feedback |
| `render_requirements` | `tree` | Converts JSON to Markdown (server-side, deterministic) for chat preview |
| `submit_requirements` | `markdown` | POSTs the final Markdown to the backend API |

The JSON is the internal format (structured, Zod-validated). The Markdown is the output: chat preview + payload sent to the backend.

## JSON Schema (Zod)

```typescript
Property          = { label: string, value: string }
RequirementNode   = {
  key: string,    // identifier: format [A-Z]{2,8}-\d{3}, e.g. WEBHOOK-001
  title: string,  // descriptive text
  properties: Property[],
  children: RequirementNode[]
}
RequirementsTree  = { version: "1.0", project_name, description, created_at, requirements: RequirementNode[] }
```

- No `id` on nodes (generated by the DB on the backend)
- `key` follows the format `[A-Z]{2,8}-\d{3}` — chosen by the LLM, meaningful prefix
- `properties` is a free array of label/value pairs — no predefined columns
- Hierarchy is free in depth

## Environment Variables

```bash

```

Copy `.env.example` → `.env`. The `submit` tool will fail gracefully if the API is not available.

## User Flow

```
[User] prompt → analyze_prompt → LLM produces the JSON tree
[User] feedback → refine_requirements → LLM refines the tree (loop)
[User] "show me" → render_requirements → Markdown displayed in chat
[User] free edits on the MD via the LLM (optional — only around tables)
[User] "submit" → submit_requirements → POST Markdown to the API
```

## Markdown Format Rules

Requirements must always appear in tables (enforced by render_requirements):
- **Vertical table** (parent): headerless table, Key in row 1, Title in row 2, properties below
- **Horizontal table** (children): Key in col 1, Title in col 2, properties as additional columns

Free Markdown (code blocks, lists, notes) can be added around the tables.
